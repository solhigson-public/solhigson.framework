# Internationalization (i18n)

Language-agnostic principles for localization and culture-aware formatting.

## Resource-Based Localization
- All user-facing text MUST use resource-based string localization (key -> translated value)
- MUST organize resource files per area or feature — MUST NOT use a single monolithic resource file
- Default language resource MUST always exist — serves as fallback when requested culture is unavailable
- Resource keys MUST be descriptive: `EventNotFound`, `WelcomeMessage` — MUST NOT use numeric or generated keys

## Culture-Aware Formatting

### Dates
- MUST use locale-appropriate date patterns (e.g., `dd/MM/yyyy` vs `MM/dd/yyyy`)
- MUST NOT hardcode date format strings — MUST use culture-aware formatting APIs

### Numbers
- MUST use locale-appropriate separators (e.g., `1,000.00` vs `1.000,00`)
- MUST NOT hardcode decimal or grouping separators

### Currency
- Currency is a business decision per transaction, NOT a display preference
- Currency symbol MUST be determined by the transaction's currency code, NOT the user's locale
- MUST NOT use locale-based currency formatting — a Nigerian user viewing a USD transaction sees `$`, not `₦`

## Culture Detection Priority
1. URL segment (e.g., `/fr/events`)
2. Cookie (user preference persisted from previous visit)
3. Accept-Language header (browser setting)
4. Default culture (application fallback)

## General Rules
- MUST NOT hardcode date, number, or currency format strings anywhere in the codebase
- MUST use culture-aware formatting APIs provided by the platform
- String concatenation for user-facing text MUST use parameterized resources, NEVER manual concatenation

## Backfill Workflow
- When adding i18n to an existing codebase, MUST extract strings incrementally by area
- MUST NOT attempt a big-bang migration — high risk of regressions and incomplete coverage
